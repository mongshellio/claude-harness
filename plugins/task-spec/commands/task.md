사용자 요청: $ARGUMENTS

위 요청을 작업하기 전에 반드시 아래 4단계 절차를 순서대로 따른다. **이 절차는 task 의 크기와 상관없이 적용된다** — trivial 한 변경(한 줄 수정 등)도 Goal/Done 두 줄은 채우고 시작한다.

`$ARGUMENTS` 가 비어 있으면, 사용자에게 "어떤 task 를 진행할지 알려달라" 고 묻고 멈춘다.

---

## 1단계 — Task Spec 작성 (Claude 가 한다)

**원칙: 사용자에게 spec 을 채우라고 시키지 않는다.** `/task` 의 존재 이유 자체가 사용자의 작성 부담을 없애는 것이다. 따라서 다음 모든 항목을 사용자의 인라인 요청 + 프로젝트 컨텍스트(CLAUDE.md, 코드 구조 등) 에서 **내가 추론해 채운다**.

- 인라인 요청에 명시되지 않은 항목도 best-guess 로 채우고 `(추정)` 으로 표기.
- 필요하면 grep / Read / `git log` 등으로 사전 조사한 뒤 채운다.
- 사용자에게 되묻는 것은 **마지막 수단**. Goal 자체가 다의적으로 해석될 때처럼 정말로 추론이 불가능한 1-2개 핵심만 묻는다.
- 5개 항목을 일일이 캐묻는 모양 금지.

### 채울 항목

- **Goal** (필수): 한 문장. 무엇을 어떤 상태로 만드는가.
- **Done = ?** (필수): 완료 판정 기준. 측정 가능 표현으로. 모호어 ("잘", "깔끔하게", "적절히", "필요시") 금지. 예시: "POST /api/foo 가 200 반환 + DB row 1개 생성 + FE 목록에 표시".
- **Non-goals**: scope 침범 차단 위해 의도적으로 안 하는 것. 사용자가 안 시킨 리팩토링·추상화·주변 정리 끼워넣기는 default 로 non-goal.
- **보호 영역**: 만지면 안 되는 파일/모듈/계층. 명시 안 됐어도 "사용자가 안 건드린 영역" 을 보수적으로 적는다.
- **참고**: 관련 CLAUDE.md / AGENTS.md / ADR / decision-log entry, 관련 코드 경로. 현재 프로젝트에 CLAUDE.md 또는 AGENTS.md 가 있다면 반드시 한 번 훑고 관련 섹션 인용.
- **영향 파일**: 변경 예상 경로 목록. 확실치 않으면 grep/Read 로 좁힌 뒤 채운다.

## 2단계 — 프로젝트 특수 룰 적용

현재 working directory 의 `CLAUDE.md` (와 sub-디렉토리 `CLAUDE.md` / `AGENTS.md`) 를 참조해, 이 task 에 발동되는 특수 규약을 spec 에 덧붙인다. 발동될 수 있는 카테고리는 프로젝트에 따라 다양하므로 아래는 예시일 뿐이다:

- 데이터 격리·접근 권한 룰 (멀티테넌트, row-level 접근 제어, ACL, 권한 매트릭스)
- 모듈·서비스·앱 경계 (어떤 책임이 어디에 속해야 하는가)
- 외부 시스템 통합 규약 (어댑터·플러그인·API 클라이언트 인터페이스 준수)
- 비동기·이벤트·큐·callback 처리 패턴
- 스키마·마이그레이션·데이터 변경 절차
- 인증·시크릿·민감 데이터 처리 규칙
- 네이밍·타입·코드 스타일 강제 규약 (linter / formatter 룰을 넘어선 도메인 컨벤션)

위 예시에 맞지 않는 프로젝트 고유 규약이 있으면 그것도 끌어와 채운다. 해당 사항 없으면 "특수 룰 해당 없음" 이라고 명시.

## 3단계 — 사용자 review

1-2단계에서 **내가 작성한 spec 초안** 을 사용자에게 보여주고 review 받는다. 사용자의 역할은 spec 작성이 아니라 **검토·승인·수정 지시** 뿐이다.

다음 형태로 한 번에 제시한다:

```
## Task Spec

**Goal**: ...
**Done**: ...
**Non-goals**: ...
**보호 영역**: ...
**참고**: ...
**영향 파일**: ...

## 프로젝트 특수 룰
- ...

## 확인 요청
이 spec 으로 진행할까요? 수정·추가·제거할 항목 있으면 알려주세요.
```

**여기서 멈춘다. 사용자의 명시적 승인 (예: "진행", "ok", "go", "수정 후 진행") 또는 수정 지시 없이 4단계로 넘어가지 않는다.** 사용자가 후속 메시지 없이 침묵하거나 다른 주제로 넘어가면 spec 은 미확정 상태로 보고 작업 시작 X.

## 4단계 — 구현

확인된 spec 범위 안에서만 작업한다. 구현 도중:

- spec 을 벗어나야 할 상황 발견 (예: Non-goals 영역 진입 필요, 보호 영역 수정 필요, 새 영향 파일 추가, Done 기준 재정의) → **즉시 일시 정지** → 사용자에게 spec 수정 제안 → 확인 후 재개
- 새로운 사실(파일 구조, 기존 코드 패턴, 라이브러리 제약 등) 로 Done 정의가 부정확해진 게 보임 → 마찬가지로 일시 정지 후 재확인
- 마무리 시 "Done 기준" 의 각 항목이 충족됐는지 한 번 명시적으로 점검 후 보고

## 안티 패턴 (절대 금지)

- spec 정리 없이 바로 코드 작성·파일 수정 시작
- **3단계 (사용자 review) 를 생략하고 1→4 로 직행** — spec 초안만 만들고 사용자 명시 승인 없이 구현 시작 금지
- **사용자에게 spec 항목들을 채워달라고 캐묻기** — 이 슬래시 커맨드의 목적 자체를 무너뜨림. 내가 추론해 채우고 사용자는 review 만.
- Goal/Done 을 모호하게 적고 진행
- 사용자가 안 시킨 리팩토링·추상화·주석/문서 추가·테스트 추가
- non-goals 영역 침범 (예: 버그 fix 요청에 주변 코드 정리 끼워넣기)
- 영향 파일을 추정하지 않고 무계획 변경
- spec 변경이 필요해졌는데 묻지 않고 임의로 확장
- 작업 끝났다고 보고할 때 Done 기준 점검 누락
